# Open-XiaoAI × Assistant

一个**有人格、有长期记忆**的多轮对话外部服务，带一个能看能改的管理前台，配合 [`examples/migpt`](../migpt) 使用。

接口协议见 [`examples/migpt/PROTOCOL.md`](../migpt/PROTOCOL.md)。

## 这是什么

改造后的 migpt 只负责**唤醒、语音识别、语音播报**，不再自己调大模型：

```
小爱固件            migpt                        assistant（本项目）
─────────           ──────────────────           ──────────────────
唤醒 + ASR   ──▶   拿到识别文字
                   打断小爱
                   POST /chat  ───────────────▶  会话上下文（内存）
                                                 灵魂 + 画像（静态注入）
                                                 search_memory（按需检索）
                                                 调用大模型
                   TTS 播报    ◀───────────────  返回文本
                   ─▶ 🔊
                                                 ↓ 异步
                                                 抽取记忆入库 + 记一笔提炼记录
```

它记得住事：说过的话会沉淀进记忆库，重启不丢；下次用到时模型自己去查。人格写在一份 Markdown 里，改文件就换性格。设计方案见 [`docs/memory.md`](docs/memory.md)。

**前台**（`http://localhost:3000`）回答的是"它怎么变成现在这样的"：

| 页面 | 看什么 / 改什么 |
| --- | --- |
| 对话与提炼 | 左边一段段列出**每次聊天**，选一段，右边就是那次的完整对话；每轮底下跟着**这轮之后它记住了什么**（新增/更新/删除了哪条）、什么都没记；另一个页签是每次巩固把**画像改成了什么**（改前改后对照）。整页 **1 秒轮询**，顶上有条实时状态：这会儿在对话、在闲置倒计时、还是在巩固画像 |
| 记忆库 | 所有明细，按类型/命中/重要度筛排，**手动删除**；最底下有**清除所有数据**（对话+记忆+画像，灵魂不动） |
| 灵魂 | 编辑 `soul.md`，存盘下一句话就生效 |
| 画像 | 编辑 `profile.md`，手写的内容不会被巩固抹掉 |

## 快速开始

```bash
cd examples/assistant

# 1. 装依赖（pnpm workspaces）
pnpm install

# 2. 起 Postgres
pnpm db:up

# 3. 配置：把原来 examples/migpt/.env 里的 OPENAI_* 搬过来
cp .env.example .env

# 4. 起前后端
pnpm dev        # api:8000 / web:3000
```

后端起来长这样：

```
✅ 外部对话服务已启动: http://0.0.0.0:8000
   模型: deepseek-v4-flash
   灵魂: data/soul.md
   会话: 最多 100 轮，闲置 300 秒算聊完（纯内存，重启即清空）
   记忆: Postgres（已载入 12 条）
   画像: data/profile.md（每轮对话结束时炼一版，另每天 03:00 兜底）
```

不接音箱也能直接调：

```bash
curl -N -X POST http://127.0.0.1:8000/chat \
  -H 'Content-Type: application/json' \
  -d '{"request_id":"1","session_id":"dev","text":"你好你是谁","stream":true}'
```

> ⚠️ **Postgres 用的是 5433 端口，不是 5432**。开发机上常常已经装了一个自己的 Postgres
> （Postgres.app、Homebrew 都默认占 5432），撞上的话轻则连不上，重则往人家库里建表。
> `docker-compose.yml` 特意错开了，`.env.example` 里的 `DATABASE_URL` 也是 5433。

## 改了表结构？把库删了重建

**这个项目没有任何迁移代码**，`migrate()`（`data/postgres.service.ts`）只保证「空库能建起来」：建表全是 `CREATE TABLE IF NOT EXISTS`，碰到已经存在的表就是空操作，你新加的列不会自己长出来。

所以改了表结构之后：

```bash
docker compose down -v   # 连数据卷一起删
pnpm db:up               # 重新建一个空库
```

灵魂和画像是文件，不在库里，删库不会碰它们。

## 和 migpt 对接

两边的配置要对上：

| assistant `.env` | migpt `.env` | 说明 |
| --- | --- | --- |
| `ASSISTANT_PORT=8000` | `AGENT_BASE_URL=http://127.0.0.1:8000` | 地址要指得对 |
| `ASSISTANT_API_KEY=xxx` | `AGENT_API_KEY=xxx` | 要一致，都留空则不校验 |

## 人格与记忆

### 心智资产放在哪

| 东西 | 存哪 | 谁写 | 为什么 |
| --- | --- | --- | --- |
| `soul.md` | **文件** | **只有你**（系统永不改写） | 灵魂是给人写、给人改的，一个编辑器就能改 |
| `profile.md` | **文件** | 巩固任务（你也可以改） | 同上，而且它要能被 diff、被备份 |
| 记忆库 | Postgres | 抽取器、巩固器 | 要无限增长、要前台按类型筛、要程序化读写 |
| 对话轮次 | Postgres | 每轮问答追加 | 要按会话翻页——全文件扫描翻到第 50 页就得读完 14 天 |
| 对话分段 | Postgres | 每轮问答顺手更新 | 段的边界只有 `SessionService` 在当时知道，见下 |
| 提炼记录 | Postgres | 抽取器、巩固器 | 要按轮次 join，见下 |

灵魂与画像刻意分离：**人格是设定的，理解是习得的**。所以「清空所有记忆」清画像、库、轮次、提炼记录，**不碰灵魂**。

> ⚠️ 用 Docker 跑的话，`data/` 要挂载卷（灵魂和画像在里面），Postgres 也要挂卷。

### 换个性格

改 `data/soul.md`，或者上前台的「灵魂」页改，**改完下一句话就生效，不用重启**（按 mtime 热加载）。两条路改的是同一份文件。文件不存在时首次启动会自动生成一份模板（活泼可爱的小管家「小蜜」）。

灵魂里只写「我是谁」，**不用写格式要求**——语音播报约束（口语短句、带标点、不要 Markdown）是传输层需求，内置在代码里，不随人格变化，也免得你改灵魂时不小心把播报弄坏。

### 记忆怎么进对话

不是把所有记忆一股脑塞进提示词（那样知识量会被预算封顶，而且越记越多、越答越笨），而是**模型自己决定什么时候查**：

```
用户：朵朵明天能去公园吗
  ↓ 模型判断需要家里的信息，发起工具调用
  🔎 search_memory: 朵朵 公园 过敏 → 2 条        （本地打分，<1ms，零网络）
  ↓ 结果作为 tool 消息进入上下文，再答一次
助手：朵朵对花粉过敏，明天公园里花粉多，要不去室内玩？
```

问「珠峰有多高」这种通用知识不会触发检索，**零额外开销**。检索触发时会多一次模型往返（约 1 秒），所以会先播一句「让我想想。」——语音场景里，这个停顿听起来就成了在回忆，而不是服务卡死。

**记忆库在 Postgres 里，但检索不查库**：启动时全量载入内存，读全走镜像，写同时落库。检索是应答路径上的一环，而「本地打分 <1ms、零网络」是这套设计的立身之本——为了一次几百微秒的字符串匹配去跑一趟 SQL，等于把工具调用那一轮省下的时间又还回去。

### 记忆怎么产生，以及它记了什么你都看得见

每轮问答落定后**异步**抽取（不占应答速度）：模型读这一轮对话，决定要 add / update / delete 哪些记忆。

抽取器的首要原则是**防污染**——闲聊、百科、一次性指令（「放首歌」）一律不记，**什么都不记是常态**。记错了比没记住更难收拾，因为错的记忆会被检索出来，然后被当真。

这个过程原来只在日志里一闪而过。现在每一次抽取和巩固都会落一条**提炼记录**，前台「对话与提炼」页直接看：左边挑一次聊天，右边是那次的完整对话，**每轮底下就跟着这轮学到了什么**。

```
┌ 6 段对话 ───┐┌ 记住，朵朵对花粉过敏 ─────────────────────────────┐
│ 记住，朵朵对 ││ 07-17 20:30 — 20:31   2 轮   default             │
│ 花粉过敏     ││                                                  │
│ 07-17 20:30 ││                            记住，朵朵对花粉过敏  │← 用户
│         2 轮 ││                                                  │
├─────────────┤│  好嘞，我记住了，朵朵对花粉过敏。                 │← 小蜜
│ 珠峰有多高   ││  ┌────────────────────────────────────────────┐  │
│ 07-17 20:12 ││  │ 记忆已更新  新增 1、更新 0、删除 0          │  │
│         1 轮 ││  │ 新增  朵朵对花粉过敏  事实        m_3bmkoc │  │
├─────────────┤│  └────────────────────────────────────────────┘  │
│ 明天几点开会 ││                                                  │
│ 07-16 09:03 ││                                    珠峰有多高    │← 用户
│         3 轮 ││                                                  │
│      …       ││  珠穆朗玛峰的海拔高度大约是八千八百四十八米……   │← 小蜜
│              ││  ┌────────────────────────────────────────────┐  │
│              ││  │ 没记  闲聊、百科、一次性指令不进记忆库      │  │
│              ││  │       ——这是常态                           │  │
│              ││  └────────────────────────────────────────────┘  │
└─────────────┘└──────────────────────────────────────────────────┘
```

「什么都没记」画得和「记了」一样清楚，因为它是**最常见的正确结果**——不显示出来，你只会以为是坏了。抽取失败（模型没按格式输出、整批丢弃）同样会记一条，带原因。

**手动删除也进这条流水**：只有模型的操作进审计、人的操作不进，那这份审计就是假的。

### 怎么让它忘

| 你说 / 你做 | 效果 |
| --- | --- |
| 「忘掉我说过不吃辣」 | 抽取器删掉对应的那条 |
| 前台记忆库页点「删除」 | 删掉那条，并记一笔「前台手动删除」 |
| 「清空所有记忆」（语音） | 画像、库、轮次、提炼记录全清，灵魂不动 |
| 前台记忆库页点「清除所有数据」 | 同上，二次确认后走同一个 `wipe()` |

「清空所有记忆」是**精确匹配整句**的，不是前缀匹配——「清空所有记忆里关于狗的部分」应该落到抽取器去删一条，不能被误杀成全量清空。

前台那个按钮在「记忆库」页最底下的危险区，语义和语音的「清空所有记忆」完全一样（同一个 `POST /memories/wipe` → `wipe()`），只是多一步二次确认。

一句话触发的不可逆操作是大忌（何况语音还会识别错），所以清空前会自动拍快照：

```bash
curl -s http://127.0.0.1:8000/memories/snapshots | jq
```

快照存在 `memory_snapshots` 表里（整个库 + 当时的画像），画像另外还会备份成 `data/profile.md-<时间戳>.bak`。巩固改写前同样拍一张。

### 越用越懂你：定时巩固

零散的记忆条目不等于「理解」。后台有一个**睡眠式**的巩固任务，把整个库和近期轮次消化一遍，做四件事：

| | 做什么 | 为什么 |
| --- | --- | --- |
| **去重合并** | 说的是同一件事的多条并成一条 | 保留最早的 `createdAt`、`hits` 相加——分散在几条上的使用次数本就是对同一件事的强化 |
| **过期清理** | 过去的日程删掉或改写成事实 | 遗忘也是记忆系统的功能。但**不许因为「很少用到」就删**：车牌号一年问一次，问起来时必须还在 |
| **模式挖掘** | 从轮次里归纳 `habit` | 单次「问天气」太琐碎，抽取器按防污染原则不记；**「每天早上都问天气」这个模式只有流水里挖得出来** |
| **画像重写** | 把全库浓缩成 `profile.md` | 泛化问题不用查库也答得合口味，还省一轮检索 |

它在两个时机跑：**每轮对话结束时**（闲置 `ASSISTANT_SESSION_TTL_SECONDS` 秒——人已经走开了，跑一次几十秒的全库调用没人等着），以及**每天 `MEMORY_CONSOLIDATE_AT`** 兜底一次（收拾等不到「对话结束」的残局，比如聊到一半重启了）。没有新记忆时**空转跳过**，不会白烧钱。

前台「画像提炼」页能看到每次巩固的**改前改后对照**，也能点「立刻巩固一次」不等它。

**画像可以手写，也不会被覆盖**：巩固的输入包含现画像全文，提示词要求保留未被新证据推翻的内容——里面可能有你亲手写的话。

### 到点自己想起来：临期日程

`dueAt` 在未来 3 天内的日程（封顶 3 条）会随请求**自动注入**，这是唯一保留的自动注入：

```
你：明天带朵朵出去玩怎么样
它：明天朵朵有钢琴课呢，上完课再出去时间会不会更宽裕？   ← 没有检索，它本来就知道
```

推送型记忆等不来检索——你问「今天有什么安排」模型会去查，但「明天有钢琴课」应该在聊到出游时**不查自知**。

### 服务商的流式工具调用不稳？换 marker

部分 OpenAI 兼容服务的流式 `tool_calls` 不稳（不支持、或 delta 格式跟标准对不上）。设 `MEMORY_RECALL_TRANSPORT=marker` 换成文本标记协议，**语义完全一样，只换传输**：模型想检索时整条回复只输出 `<搜记忆:朵朵 过敏>`，服务端嗅探首块拦下来，走同一个检索执行器、同样的二次调用、同样的次数上限。

常规回答零影响：首字符不是 `<` 立即透传，不会为了等一个可能不存在的标记把整句话卡在缓冲区里。

## 设计要点

### 会话窗口是纯内存的，长期记忆才落库

最近几轮对话存在一个 `Map<session_id, Session>` 里，**没有任何持久化，进程重启即清空**——即使记忆库已经进了 Postgres，这里依然是内存。会话窗口是「这次对话的流」，不是记忆：值得留下的东西每轮都会被抽取进长期记忆，窗口本身没有留存价值。

这也正是**每轮都要当场抽取记忆**的原因：会话在内存里、有 TTL、随时可能重启，经历转瞬即逝，不能等一个定时任务来收。

闲置到期这个时刻还有第二个用处：**它是「用户安静下来了」的唯一信号**，巩固任务拿它当睡眠钟。所以这里用真定时器而不是懒惰过期——懒惰过期下「对话结束」这个时刻根本不存在，会话是在用户下次开口时才被清掉的，那时人已经在说下一句了。

**闲置只算「静默」，不算「处理」**：一轮问答（模型思考 + 检索 + 语音合成）可能比 TTL 还久——尤其 TTL 调得短、又用了思考模式的模型时。这段时间里没有任何 `touch()`，定时器要是照走，会话就在这一轮还没答完时当场过期：下一句被切成新的一段对话，还会在这轮的抽取都没跑完时就误触发一次巩固（抽出来的记忆赶不上，画像就漏了）。所以 `SessionService` 给每个会话记了个 `inFlight` 计数，请求在处理中就不计闲置，等这一轮**彻底做完**（含落库、进抽取队列）才从头开始倒计时。改会话过期逻辑时别把这个前提破坏了。

**巩固触发不能丢**：`onIdle` 触发巩固时，如果上一次巩固还在跑，不能直接早退把这次触发扔掉——慢一轮抽出来的记忆常常赶不上正在跑的那班巩固。正确做法是记一笔、炼完补一次（多个触发合并成一次）。早退丢触发的话，那条记忆就只能等明天凌晨那趟兜底了。

`session_id` 由 migpt 侧的 `AGENT_SESSION_ID` 决定，默认 `default`。长期记忆和人格是**全局一份**，不按 `session_id` 隔离——`session_id` 是对话的边界，不是人的边界。

### 一次对话 = 一个会话窗口的生命周期

「一次对话」不能靠 `session_id` 分：协议里没有唤醒边界，`session_id` 由 migpt 那边固定给（默认所有对话都是同一个 `default`），拿它当对话列表只会看到孤零零一项——十天前问天气和刚才聊朵朵会挤在一起。

段的边界就是**会话窗口的生死**，这件事 `SessionService` 本来就知道，不用谁去猜：

```
用户开口 ──▶ 建会话窗口，顺手生成 c_lz4k8f3a   ┐
             每轮问答都带着这个 id 落进 turns   │ 一次对话
闲置 TTL ──▶ 窗口过期（清上下文 + 炼画像）      ┘
用户再开口 ─▶ 建新窗口，新 id c_lz4kb1x8        ← 新的一段
```

id 在**建窗口那一刻**就生成，每轮落库时一并写进 `turns.conversation_id`，同时在 `conversations` 表里把轮数和结束时间往前推。所以段的归属是**当时定下的事实**，不是事后拿时间戳猜的——这也意味着之后再改 `ASSISTANT_SESSION_TTL_SECONDS`，已经存下来的历史不会被重新切一遍。

前台读 `conversations` 表就是一次普通的翻页查询，不做任何聚合和推导。

两个容易踩的点：

- **段在用户开口时就取好，不能等模型答完再取**。大模型可能跑得比 TTL 还久，那时窗口已经过期重建，这一轮就会被算进一段它根本不属于的新对话。抽取是异步的，同理——`Round` 里带着 `conversationId` 一路传到落库。
- **「重新开始」也会开新的一段**。`reset()` 把窗口删了，下次开口就是新窗口、新 id。这是对的：用户说「重新开始」，那就是另一次对话了。

> TTL 调得很短（比如调成 10 秒来调试）时，对话会切得很碎。**这不是显示错了**——服务端当时确实把它们当成了一段段独立的对话，每段结束都清了上下文、炼了画像。

### 前台不碰密钥

浏览器只访问前台自己的 `/api/*`，由前台的**服务端**代理转发时补上 `Authorization`（见 `apps/web/app/api/[...path]/route.ts`）。`ASSISTANT_API_KEY` 和 `ASSISTANT_API_URL` 都不带 `NEXT_PUBLIC_` 前缀，进不了浏览器。

代理走**白名单**而不是全放开：它会自动补密钥，等于谁能访问前台谁就有后端的完整权限——至少别把 `/chat` 也捎上，那是音箱走的路，从浏览器发起会污染会话上下文和记忆。

### 播报约束内置在代码里

migpt 侧按 `。？！；?!;` 分句后逐句合成语音，**一段没有标点的长回复在流结束前一个字都播不出来**。所以「口语短句、正常标点、不要 Markdown」这条约束写死在 `soul/soul.service.ts` 里，和人格分开，`soul.md` 里不用也不该写它。

### 语义交给模型，代码只做 join

打分器是纯字符串匹配，看着很笨，但它不是朴素的关键词检索——**语义活儿在两头已经被模型干完了**：

```
写入时：抽取器把语义压成 subjects/keywords，还负责挂别称（"朵朵"→"女儿""闺女"）
读取时：模型读完整个上下文，生成检索词
本地打分：只负责把两头 join 起来，<1ms、零网络
```

举个最容易踩的例子：没有说话人标识，所以「我」进了库一律记成 `subjects: ["用户"]`。那你问「我是谁」，检索词里只有「我」，跟 `["用户"]` 一个字都对不上，岂不是必然落空？

**不会——因为翻译这一步是模型做的**。工具说明里教了它「问『我……』就用『用户』当人称」，实测各种问法（我是谁 / 我叫啥 / 俺叫什么来着）它都稳定生成 `用户 名字`。

所以**别在打分器里加同义词表、代词映射这类硬编码**：模型有完整上下文，比字符串匹配清楚得多。规则写进提示词，打分器保持又笨又快又可预测——出了问题，你能一眼看出为什么没命中。

### 工具循环在服务内部闭环

`/chat` 的请求和响应格式**一个字都没变**：`search_memory` 的工具调用、tool 消息、二次请求全在服务内部完成，migpt 只看到一条正常的 SSE 文本流。工具消息也不进会话历史——这一轮的答案已经把检索内容吸收了，跨轮靠答案传递。

带检索的模型往返有次数上限（`MEMORY_SEARCH_MAX_CALLS`），到了上限就把 tools 摘掉逼模型作答。数的是**往返轮次**而不是检索条数：模型可能在一轮里并行查好几个词（「车牌号」+「生日」+「手机号」），那也只是一次往返——真正要防的是循环，而检索本身在本地跑，不要钱。

### 历史按「轮」成对淘汰

提问和回复成对写入、成对淘汰。如果按「条」淘汰，窗口边缘会留下没有提问的孤儿回复，白占一个槽位。

### 失败的请求不污染上下文

发给大模型的消息是临时拼的（`system + 历史 + 本次提问`），**只有成功拿到回复后才把这一轮成对写入历史**。所以调用失败、鉴权失败都不会在上下文里留下垃圾。

### 抢话打断会记录已生成的部分

用户抢话时 migpt 会直接断开 HTTP 连接，本服务据此 abort 大模型请求，并把**已经生成的那部分回复**记进历史。否则上下文里会留下一条没有回复的提问，下一轮就变成连续两条 `user` 消息。

需要注意的是：由于语音合成比生成慢，**记进历史的部分会比用户实际听到的多一些**。这是个已知的近似。

这一轮照样会落轮次、进抽取队列：用户说的话在 `user` 消息里，不受回复被截断影响——「记住我对海鲜过敏」说完就被抢话，也得记住。

### 写操作全部串行

抽取套用、轮次追加、命中回写、清空都经 `MemoryService` 同一条 promise 链串行执行，和 `SessionService.lock()` 相互独立。落库时整批在一个事务里——模型的一批操作往往是有关联的（先 update 再 delete），只成一半会让库处于中间状态。

清空时要**连抽取队列的积压一起清掉**，否则清空之前那几轮的抽取结果会在清空之后落库，刚被删掉的信息就「复活」了。

### 同一会话的请求串行执行

用户抢话时，migpt 会先断开上一条请求再发新的，两条请求可能在服务端交叠。`SessionService.lock()` 让同一 `session_id` 的请求排队执行，否则历史会写成乱序。

## 已知取舍

1. **记忆是按「这台音箱」记的，不按人**。`/chat` 的入参只有 `text`，没有说话人标识，所以谁说的都记成「用户」。缓解办法是自报身份（「我是爸爸，我不吃辣」），抽取器会把它写进 `subjects`。
2. **语音识别的错字可能被记成错误的记忆**。抽取提示词里要求「拿不准宁可不记」，每条记忆也留了 `evidence` 原话摘录可供审计（前台记忆库页直接显示）；口头更正会走 update 改掉它。
3. **隐私是明文的**。灵魂和画像明文躺在磁盘上，记忆和对话原文明文躺在 Postgres 里，画像和检索结果会随请求发给大模型服务商。**前台没有登录**，谁能访问它谁就能看到全部记忆——别把它暴露到公网。不想留对话原文就设 `MEMORY_TRANSCRIPT_DAYS=0`。
4. **检索依赖模型自觉**。该查不查就会答出错误的「不知道」，不该查乱查就白付一轮往返。系统提示词里两个方向都做了约束。
5. **流式工具调用的兼容性参差** → 换 `MEMORY_RECALL_TRANSPORT=marker`，语义不变。但要注意：`marker` 是我们自己约定的文本协议，不像 function calling 那样是模型训练过的行为，**小模型上更容易不听话**。能用 `tools` 就用 `tools`。
6. **`hits` 是代理信号**。被检索到 ≠ 被用上，只作排序参考，不决定存废。
7. **巩固是一次全库调用**，几十秒起步。它跑在后台、不占应答，但库很大时要留意 token 成本。
8. **内存镜像和库之间有一个很短的不一致窗口**。落库失败时会从库里重新载入镜像（宁可丢掉这批刚抽出来的记忆，也不让两边长期各说各话），但那一瞬间检索读到的可能是还没落库的内容。

## 换成你自己的 Agent

本服务只做一件事：**收文本 → 出文本**。把 `chat/chat.service.ts` 里 `complete()` 中调用 `this.llm` 的部分换成你自己的逻辑即可，协议层、会话管理、取消处理都不用动。

如果你的 Agent 自带上下文管理，可以直接把 `SessionService` 摘掉，用 `session_id` 关联到你自己的会话。同理，不需要记忆和人格的话，设 `MEMORY_ENABLED=false` 即可回到纯内存版。
