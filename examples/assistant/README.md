# Open-XiaoAI × Assistant

一个**有人格、有长期记忆**的多轮对话外部服务，配合 [`examples/migpt`](../migpt) 使用。

接口协议见 [`examples/migpt/PROTOCOL.md`](../migpt/PROTOCOL.md)。

## 这是什么

改造后的 migpt 只负责**唤醒、语音识别、语音播报**，不再自己调大模型：

```
小爱固件            migpt                        assistant（本项目）
─────────           ──────────────────           ──────────────────
唤醒 + ASR   ──▶   拿到识别文字
                   打断小爱
                   POST /chat  ───────────────▶  会话上下文（内存）
                                                 灵魂 + 画像（静态注入）
                                                 search_memory（按需检索）
                                                 调用大模型
                   TTS 播报    ◀───────────────  返回文本
                   ─▶ 🔊
                                                 ↓ 异步
                                                 抽取记忆入库
```

它记得住事：说过的话会沉淀进本地记忆库，重启不丢；下次用到时模型自己去查。人格写在一份 Markdown 里，改文件就换性格。设计方案见 [`docs/memory.md`](docs/memory.md)。

原来写在 `examples/migpt/src/config.ts` 里的大模型配置（`OPENAI_*`、系统提示词、历史轮数）全部搬到了这里。

## 快速开始

```bash
cd examples/assistant

# 1. 装依赖
pnpm install

# 2. 配置：把原来 examples/migpt/.env 里的 OPENAI_* 搬过来
cp .env.example .env

# 3. 启动
pnpm start
```

启动后：

```
✅ 外部对话服务已启动: http://0.0.0.0:8000
   模型: deepseek-v4-flash
   灵魂: data/soul.md
   会话: 10 轮（纯内存，重启即清空）
   记忆: data/memory.json（已载入 0 条）
```

不接音箱也能直接调：

```bash
curl -N -X POST http://127.0.0.1:8000/chat \
  -H 'Content-Type: application/json' \
  -d '{"request_id":"1","session_id":"dev","text":"你好你是谁","stream":true}'
```

## 和 migpt 对接

两边的配置要对上：

| assistant `.env` | migpt `.env` | 说明 |
| --- | --- | --- |
| `ASSISTANT_PORT=8000` | `AGENT_BASE_URL=http://127.0.0.1:8000` | 地址要指得对 |
| `ASSISTANT_API_KEY=xxx` | `AGENT_API_KEY=xxx` | 要一致，都留空则不校验 |

migpt 侧的 `OPENAI_*` 可以全部删掉（内置大模型已被绕过），`TTS_*` 必须保留。

先起 assistant 再起 migpt。migpt 启动时会探测一次 `/health`，探测失败只告警不阻塞。

## 配置

全部在 `.env` 里，对应 `config.ts`。

**大模型**（和原来 migpt 的配置完全一样，直接搬）：

| 配置 | 说明 |
| --- | --- |
| `OPENAI_API_KEY` | 密钥 |
| `OPENAI_BASE_URL` | 接口地址，一般以 `/v1` 结尾 |
| `OPENAI_MODEL` | 模型名称 |
| `OPENAI_TEMPERATURE` | 温度，删除则用服务商默认值 |
| `OPENAI_THINKING` | 思考模式，DeepSeek 参数格式，其他服务商可能要删掉 |

**多轮对话**（会话窗口，纯内存）：

| 配置 | 默认值 | 说明 |
| --- | --- | --- |
| `ASSISTANT_MAX_TURNS` | 10 | 最多记住多少**轮**（一轮 = 一问一答） |
| `ASSISTANT_SESSION_TTL_MS` | 1800000 | 会话闲置多久后清空 |
| `ASSISTANT_RESET_KEYWORDS` | 重新开始,忘掉刚才 | 说这些话可以清空**会话上下文** |

> ⚠️ **默认值变更**：`ASSISTANT_RESET_KEYWORDS` 原来是 `重新开始,清空记忆,忘掉刚才`。引入长期记忆后「清空记忆」有歧义（清哪个？），已从默认值里移除。清长期记忆请说「清空所有记忆」，见下面的 `MEMORY_WIPE_KEYWORDS`。

**人格**：

| 配置 | 默认值 | 说明 |
| --- | --- | --- |
| `ASSISTANT_SOUL_FILE` | data/soul.md | 灵魂文件，**不受 `MEMORY_ENABLED` 影响** |
| `ASSISTANT_SYSTEM_PROMPT` | （空） | **已废弃**。填了会整体替换「灵魂 + 播报约束」，`soul.md` 就不生效了 |

**长期记忆**：

| 配置 | 默认值 | 说明 |
| --- | --- | --- |
| `MEMORY_ENABLED` | true | 总开关，`false` 回到纯内存版（灵魂照常生效） |
| `MEMORY_FILE` | data/memory.json | 记忆库 |
| `MEMORY_PROFILE_FILE` | data/profile.md | 画像（目前只能手写，见下） |
| `MEMORY_TRANSCRIPT_FILE` | data/transcript.jsonl | 对话流水 |
| `MEMORY_TRANSCRIPT_DAYS` | 14 | 流水保留天数，`0` 则不落流水 |
| `MEMORY_RECALL_TOP_K` | 8 | 单次检索最多返回多少条 |
| `MEMORY_RECALL_MAX_CHARS` | 1500 | 单次检索结果的长度预算 |
| `MEMORY_SEARCH_MAX_CALLS` | 2 | 每个请求最多允许几**轮**带检索的模型往返 |
| `MEMORY_SEARCH_FILLER` | 让我想想。 | 检索时先播的填补话术，置空关闭 |
| `MEMORY_RECALL_TRANSPORT` | tools | `tools`（标准 function calling）\| `marker`（文本标记协议） |
| `MEMORY_PROFILE_MAX_CHARS` | 1000 | 画像预算 |
| `MEMORY_CONSOLIDATE_INTERVAL_HOURS` | 24 | 每隔多久检查执行一次巩固 |
| `MEMORY_CONSOLIDATE_EVERY_N` | 50 | 攒够多少条变更就提前巩固（兜底触发） |
| `MEMORY_WIPE_KEYWORDS` | 清空所有记忆 | 说这句话清空**长期记忆**（精确匹配整句） |
| `MEMORY_WIPE_TEXT` | 好的，我已经把记住的事情都忘掉了。 | 清空后的回复话术 |
| `MEMORY_OPENAI_*` | 复用 `OPENAI_*` | 抽取/巩固专用模型，可以配个便宜的 |

## 人格与记忆

### 四份文件

助手的「心智资产」全部落在 `data/` 目录，可备份、可迁移、可直接用编辑器改：

| 文件 | 谁写 | 角色 |
| --- | --- | --- |
| `soul.md` | **只有你**（系统永不改写） | **灵魂**：性格、说话风格、自称、边界 |
| `profile.md` | 定时巩固任务（你也可以改） | **画像**：对这个家的理解 |
| `memory.json` | 抽取器 | **记忆库**：结构化明细，可无限增长 |
| `transcript.jsonl` | 每轮问答追加 | **流水**：近期原始对话 |

灵魂与画像刻意分离：**人格是设定的，理解是习得的**。所以「清空所有记忆」清画像、库、流水，**不碰灵魂**。

> ⚠️ 用 Docker 跑的话，`data/` **必须挂载卷**，否则容器一重建这四份文件全丢。

### 换个性格

改 `data/soul.md` 就行，**改完下一句话就生效，不用重启**（按 mtime 热加载）。文件不存在时首次启动会自动生成一份模板（活泼可爱的小管家「小蜜」）。

灵魂里只写「我是谁」，**不用写格式要求**——语音播报约束（口语短句、带标点、不要 Markdown）是传输层需求，内置在代码里，不随人格变化，也免得你改灵魂时不小心把播报弄坏。

### 记忆怎么进对话

不是把所有记忆一股脑塞进提示词（那样知识量会被预算封顶，而且越记越多、越答越笨），而是**模型自己决定什么时候查**：

```
用户：朵朵明天能去公园吗
  ↓ 模型判断需要家里的信息，发起工具调用
  🔎 search_memory: 朵朵 公园 过敏 → 2 条        （本地打分，<1ms，零网络）
  ↓ 结果作为 tool 消息进入上下文，再答一次
助手：朵朵对花粉过敏，明天公园里花粉多，要不去室内玩？
```

问「珠峰有多高」这种通用知识不会触发检索，**零额外开销**。检索触发时会多一次模型往返（约 1 秒），所以会先播一句「让我想想。」——语音场景里，这个停顿听起来就成了在回忆，而不是服务卡死。

### 记忆怎么产生

每轮问答落定后**异步**抽取（不占应答速度）：模型读这一轮对话，决定要 add / update / delete 哪些记忆。

抽取器的首要原则是**防污染**——闲聊、百科、一次性指令（「放首歌」）一律不记，**什么都不记是常态**。记错了比没记住更难收拾，因为错的记忆会被检索出来，然后被当真。

```bash
# 看看它都记了些什么
curl -s http://127.0.0.1:8000/memories | jq
```

这条学习闭环是它「越用越智能」的地方：**对话 → 当场编码 → 被检索用到 → 强化排序（`hits`）→ 定时巩固成画像与习惯 → 反哺下一次回答**。

### 怎么让它忘

| 你说 | 效果 |
| --- | --- |
| 「忘掉我说过不吃辣」 | 抽取器删掉对应的那条 |
| 「清空所有记忆」 | 画像、库、流水全清，灵魂不动 |

「清空所有记忆」是**精确匹配整句**的，不是前缀匹配——「清空所有记忆里关于狗的部分」应该落到抽取器去删一条，不能被误杀成全量清空。

一句话触发的不可逆操作是大忌（何况语音还会识别错），所以清空前会自动备份：

```bash
ls data/*.bak          # memory.json-2026-07-17T...bak 之类
```

### 越用越懂你：定时巩固

零散的记忆条目不等于「理解」。后台有一个**睡眠式**的巩固任务（默认每 24 小时检查一次，攒够 50 条变更会提前跑），把整个库和近期流水消化一遍，做四件事：

| | 做什么 | 为什么 |
| --- | --- | --- |
| **去重合并** | 说的是同一件事的多条并成一条 | 保留最早的 `createdAt`、`hits` 相加——分散在几条上的使用次数本就是对同一件事的强化 |
| **过期清理** | 过去的日程删掉或改写成事实；长期没被用过的琐碎条目淘汰 | 遗忘也是记忆系统的功能，不然库越大噪声越大 |
| **模式挖掘** | 从流水里归纳 `habit` | 单次「问天气」太琐碎，抽取器按防污染原则不记；**「每天早上都问天气」这个模式只有流水里挖得出来** |
| **画像重写** | 把全库浓缩成 `profile.md` | 泛化问题不用查库也答得合口味，还省一轮检索 |

没有新变化时会**空转跳过**，不会白烧钱。改写前 `memory.json` 和 `profile.md` 都会先备份，任何一步没通过校验就整体保持原状。

**画像可以手写，也不会被覆盖**：巩固的输入包含现画像全文，提示词要求保留未被新证据推翻的内容——里面可能有你亲手写的话。

```bash
# 平时是天级后台跑的，想立刻看效果：
curl -s -X POST http://127.0.0.1:8000/memories/consolidate | jq -r .profile
```

跑完大概长这样：

```markdown
# 称呼与家庭
用户是两口之家加一个女儿朵朵。

# 习惯与作息
用户每天早晨七点左右会查询天气并播放音乐。      ← 从流水里挖出来的

# 近期安排
明天（7月18日）朵朵有钢琴课；用户计划8月16日去北京出差。   ← 从 dueAt 条目生成
```

### 到点自己想起来：临期日程

`dueAt` 在未来 3 天内的日程（封顶 3 条）会随请求**自动注入**，这是唯一保留的自动注入：

```
你：明天带朵朵出去玩怎么样
它：明天朵朵有钢琴课呢，上完课再出去时间会不会更宽裕？   ← 没有检索，它本来就知道
```

推送型记忆等不来检索——你问「今天有什么安排」模型会去查，但「明天有钢琴课」应该在聊到出游时**不查自知**。

### 服务商的流式工具调用不稳？换 marker

部分 OpenAI 兼容服务的流式 `tool_calls` 不稳（不支持、或 delta 格式跟标准对不上）。设 `MEMORY_RECALL_TRANSPORT=marker` 换成文本标记协议，**语义完全一样，只换传输**：模型想检索时整条回复只输出 `<搜记忆:朵朵 过敏>`，服务端嗅探首块拦下来，走同一个检索执行器、同样的二次调用、同样的次数上限。

常规回答零影响：首字符不是 `<` 立即透传，不会为了等一个可能不存在的标记把整句话卡在缓冲区里。

## 设计要点

### 会话窗口是纯内存的，长期记忆才落盘

最近几轮对话存在一个 `Map<session_id, Session>` 里，**没有任何持久化，进程重启即清空**。闲置超过 `ASSISTANT_SESSION_TTL_MS` 的会话会在下次请求时被回收，避免 Map 无限增长。

这也正是**每轮都要当场抽取记忆**的原因：会话在内存里、有 TTL、随时可能重启，经历转瞬即逝，不能等一个定时任务来收。

`session_id` 由 migpt 侧的 `AGENT_SESSION_ID` 决定，默认 `default`。migpt 同一时刻只服务一个音箱，所以单实例场景下就是一个固定会话。

长期记忆和人格则是**全局一份**，不按 `session_id` 隔离——`session_id` 是对话的边界，不是人的边界。

### 播报约束内置在代码里

migpt 侧按 `。？！；?!;` 分句后逐句合成语音，**一段没有标点的长回复在流结束前一个字都播不出来**。所以「口语短句、正常标点、不要 Markdown」这条约束写死在 `soul.ts` 里，和人格分开，`soul.md` 里不用也不该写它。

### 语义交给模型，代码只做 join

打分器是纯字符串匹配，看着很笨，但它不是朴素的关键词检索——**语义活儿在两头已经被模型干完了**：

```
写入时：抽取器把语义压成 subjects/keywords，还负责挂别称（"朵朵"→"女儿""闺女"）
读取时：模型读完整个上下文，生成检索词
本地打分：只负责把两头 join 起来，<1ms、零网络
```

举个最容易踩的例子：没有说话人标识，所以「我」进了库一律记成 `subjects: ["用户"]`。那你问「我是谁」，检索词里只有「我」，跟 `["用户"]` 一个字都对不上，岂不是必然落空？

**不会——因为翻译这一步是模型做的**。工具说明里教了它「问『我……』就用『用户』当人称」，实测各种问法（我是谁 / 我叫啥 / 俺叫什么来着 / 咱这个家谁做主）它都稳定生成 `用户 名字`，甚至会直接用上真名 `周涛 车牌号`。

所以**别在打分器里加同义词表、代词映射这类硬编码**：模型有完整上下文，比字符串匹配清楚得多。规则写进提示词，打分器保持又笨又快又可预测——出了问题，你能一眼看出为什么没命中。

代价是检索准不准依赖模型自觉（见已知取舍）。真要补，也该补在提示词或抽取的别称上，而不是在这里堆规则。

### 工具循环在服务内部闭环

`/chat` 的请求和响应格式**一个字都没变**：`search_memory` 的工具调用、tool 消息、二次请求全在服务内部完成，migpt 只看到一条正常的 SSE 文本流。工具消息也不进会话历史——这一轮的答案已经把检索内容吸收了，跨轮靠答案传递。

带检索的模型往返有次数上限（`MEMORY_SEARCH_MAX_CALLS`），到了上限就把 tools 摘掉逼模型作答，防止它查完再查停不下来。数的是**往返轮次**而不是检索条数：模型可能在一轮里并行查好几个词（「车牌号」+「生日」+「手机号」），那也只是一次往返——真正要防的是循环，而检索本身在本地跑，不要钱。

### 历史按「轮」成对淘汰

提问和回复成对写入、成对淘汰。如果按「条」淘汰，窗口边缘会留下没有提问的孤儿回复，白占一个槽位。

### 失败的请求不污染上下文

发给大模型的消息是临时拼的（`system + 历史 + 本次提问`），**只有成功拿到回复后才把这一轮成对写入历史**。所以调用失败、鉴权失败都不会在上下文里留下垃圾。

### 抢话打断会记录已生成的部分

用户抢话时 migpt 会直接断开 HTTP 连接，本服务据此 abort 大模型请求，并把**已经生成的那部分回复**记进历史。

否则上下文里会留下一条没有回复的提问，下一轮就变成连续两条 `user` 消息。

需要注意的是：由于语音合成比生成慢，**记进历史的部分会比用户实际听到的多一些**。这是个已知的近似。

这一轮照样会落流水、进抽取队列：用户说的话在 `user` 消息里，不受回复被截断影响——「记住我对海鲜过敏」说完就被抢话，也得记住。

### 写操作全部串行

抽取套用、流水追加、命中回写、清空都经 `MemoryManager` 同一条 promise 链串行执行，和 `SessionStore.lock()` 相互独立。

记忆库的每次变更都是**原子写**（先写临时文件再 `rename`）：直接写目标文件的话，写到一半崩溃会留下半截 JSON，下次启动整个库都没了。

清空时要**连抽取队列的积压一起清掉**，否则清空之前那几轮的抽取结果会在清空之后落库，刚被删掉的信息就「复活」了。

### 同一会话的请求串行执行

用户抢话时，migpt 会先断开上一条请求再发新的，两条请求可能在服务端交叠。`SessionStore.lock()` 让同一 `session_id` 的请求排队执行，否则历史会写成乱序。

## 已知取舍

1. **记忆是按「这台音箱」记的，不按人**。`/chat` 的入参只有 `text`，没有说话人标识，所以谁说的都记成「用户」。缓解办法是自报身份（「我是爸爸，我不吃辣」），抽取器会把它写进 `subjects`。
2. **语音识别的错字可能被记成错误的记忆**。抽取提示词里要求「拿不准宁可不记」，每条记忆也留了 `evidence` 原话摘录可供审计；口头更正会走 update 改掉它。
3. **隐私是明文的**。四份文件明文躺在本地磁盘，画像和检索结果会随请求发给大模型服务商。敏感信息请自行斟酌，不想留对话原文就设 `MEMORY_TRANSCRIPT_DAYS=0`。
4. **检索依赖模型自觉**。该查不查就会答出错误的「不知道」，不该查乱查就白付一轮往返。系统提示词里两个方向都做了约束。
5. **流式工具调用的兼容性参差** → 换 `MEMORY_RECALL_TRANSPORT=marker`，语义不变。但要注意：`marker` 是我们自己约定的文本协议，不像 function calling 那样是模型训练过的行为，**小模型上更容易不听话**（该输标记时不输，直接开始瞎编）。能用 `tools` 就用 `tools`。
6. **`hits` 是代理信号**。被检索到 ≠ 被用上，只作排序参考，不决定存废。
7. **巩固是一次全库调用**，几十秒起步。它跑在后台、不占应答，但库很大时要留意 token 成本。

## 目录结构

```
src/
  soul.ts            # 灵魂/画像装载（mtime 热更新）、静态区拼装、灵魂模板
  memory/
    types.ts         # 共享类型
    store.ts         # 记忆库：载入、原子落盘、CRUD、合并、操作校验
    transcript.ts    # 流水：追加、按天清理
    search.ts        # 检索执行器：本地打分（纯函数、零 IO）
    extractor.ts     # 每轮抽取：提示词与调用、JSON 解析校验
    consolidator.ts  # 定时巩固：去重、过期清理、模式挖掘、画像重写
    marker.ts        # 文本标记协议与流式嗅探器
    manager.ts       # 对外唯一门面：单写者队列、抽取队列、巩固触发、清空
  llm.ts             # 大模型调用，支持 tools 与流式 tool_call 聚合
  server.ts          # HTTP + 检索循环编排（tools / marker 两种传输）
  session.ts         # 会话窗口（纯内存）
  config.ts          # 配置
```

HTTP 接口：`POST /chat`（协议不变）、`GET /health`、`GET /memories`、`POST /memories/consolidate`。

## 换成你自己的 Agent

本服务只做一件事：**收文本 → 出文本**。把 `server.ts` 里 `complete()` 中调用 `this.llm` 的部分换成你自己的逻辑即可，协议层、会话管理、取消处理都不用动。

如果你的 Agent 自带上下文管理，可以直接把 `SessionStore` 摘掉，用 `session_id` 关联到你自己的会话。同理，不需要记忆和人格的话，设 `MEMORY_ENABLED=false` 即可回到纯内存版。
